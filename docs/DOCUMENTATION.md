# Alfy² — Documentation Structure

How documentation itself is organized, so it stays trustworthy as the system grows.

---

## 1. Where docs live
- **`/README.md`** — the front door; orientation + quick start + links out.
- **`/ARCHITECTURE.md`** — the durable system shape (changes rarely, via ADR).
- **`/docs/*`** — the working knowledge base (below).
- **Inline code docs** — only in `packages/shared` for contracts; elsewhere comment the *why*.

## 3. Rules
- **Docs ship with code.** A structural or behavioral change updates its doc in the *same* PR.
- **One source of truth.** Don't restate a fact in two docs; link instead.
- **ADRs are append-only.** Decisions are superseded by new ADRs, never edited away.
- **Glossary is law.** Use glossary terms verbatim in code and docs.
- **Keep it prose + tables.** Avoid deep nesting; a doc should be skimmable in two minutes.

## 4. Adding documentation
1. If it's a decision → new `ADR-NNNN`.
2. If it's a standard/process → extend the matching `docs/*` file (don't create overlap).
3. If it's feature behavior (later phases) → a `docs/modules/<name>.md` page, linked from the module manifest.
4. Always add new docs to the table above and to `FOLDER_HIERARCHY.md`.
